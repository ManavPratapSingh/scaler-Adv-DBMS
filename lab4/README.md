# SQLite3 Database Internal Structure Analysis using XXD

## Objective

In this lab we have to analyze the internal structure of a real SQLite3 database file using hexadecimal dumps generated by the `xxd` utility. The experiment shows how SQLite internally stores:

- Database headers
- Page metadata
- B-tree pages
- Table records
- Cell pointer arrays
- B-tree child pointers
- Rowid lookup paths
- Schema definitions

## Files Submitted

- `students.db` - real SQLite3 database created for this lab.
- `hexdump.txt` - real hex dump generated using `xxd`.
- `README.md` - explanation of SQLite page navigation, B-tree nodes, lookup, and addresses.

SQLite database files begin with a 100-byte database header on page 1.

Real `xxd` output from `hexdump.txt`:

```text

00000010: 02 00 01 01 0c 40 20 20 00 00 00 03 00 00 00 13  .....@  ........
00000020: 00 00 00 00 00 00 00 00 00 00 00 02 00 00 00 04  ................
00000030: 00 00 00 00 00 00 00 00 00 00 00 01 00 00 00 00  ................
00000040: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00  ................
00000050: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 03  ................
00000060: 00 2e 8d f8 0d 00 00 00 01 01 34 00 01 34 00 00  ..........4..4..
```

Important header fields:

| Offset | Bytes | Meaning |
|---:|---|---|
| `0x0000` | `53 51 4c 69 74 65 20 66 6f 72 6d 61 74 20 33 00` | SQLite signature: `SQLite format 3\0` |
| `0x0010` | `02 00` | Page size = `0x0200` = 512 bytes |
| `0x001c` | `00 00 00 13` | Database page count = `0x13` = 19 pages |
| `0x0064` | `0d` | Page 1 B-tree page type = leaf table page |

For this file:

```text
page_size = 512 bytes
page_count = 19
file_offset(page N) = (N - 1) * 512
```

## SQLite Header Breakdown

| Bytes | Meaning |
|---|---|
| `53 51 4c 69` | `SQLi` |
| `74 65` | `te` |
| `20 66 6f 72 6d 61 74` | ` format` |
| `20 33 00` | ` 3\0` |

This is the SQLite magic header that identifies the file as a valid SQLite3 database.

## SQLite B-tree Page Header Format

Every B-tree page has a page header.

For page 1, the B-tree header starts at file offset `0x0064`, because the first 100 bytes are the database header.

For all other pages, the B-tree header starts at the beginning of that page.

| Page type byte | Meaning |
|---:|---|
| `0x05` | Interior table B-tree page |
| `0x0d` | Leaf table B-tree page |

Table B-tree page header fields:

| Header offset | Size | Meaning |
|---:|---:|---|
| `+0` | 1 byte | Page type |
| `+1` | 2 bytes | First freeblock offset |
| `+3` | 2 bytes | Number of cells |
| `+5` | 2 bytes | Start of cell content area |
| `+7` | 1 byte | Fragmented free bytes |
| `+8` | 4 bytes | Right-most child pointer, only on interior pages |

After the page header comes the cell pointer array. Each cell pointer is 2 bytes and stores the page-relative offset of one cell.

## Page 2: Root Interior Table B-tree Page

Page 2 starts at:

```text
(2 - 1) * 512 = 512 = 0x0200
```

Real hex dump:

```text
00000200: 05 00 00 00 10 01 9d 00 00 00 00 13 01 ef 01 ea  ................
00000210: 01 e5 01 e0 01 db 01 d6 01 d1 01 cc 01 c7 01 c1  ................
00000220: 01 bb 01 b5 01 af 01 a9 01 a3 01 9d 00 00 00 00  ................
```

Page 2 header decoding:

| File offset | Bytes | Meaning |
|---:|---|---|
| `0x0200` | `05` | Interior table B-tree page |
| `0x0201` | `00 00` | No freeblock |
| `0x0203` | `00 10` | 16 cells |
| `0x0205` | `01 9d` | Cell content starts at page offset `0x019d` |
| `0x0207` | `00` | 0 fragmented bytes |
| `0x0208` | `00 00 00 13` | Right-most child page = page 19 |

The cell pointer array starts at `0x020c`. It contains 16 two-byte pointers:

```text
0x01ef, 0x01ea, 0x01e5, 0x01e0,
0x01db, 0x01d6, 0x01d1, 0x01cc,
0x01c7, 0x01c1, 0x01bb, 0x01b5,
0x01af, 0x01a9, 0x01a3, 0x019d
```

An interior table B-tree cell stores:

```text
4-byte left child page number + varint separator rowid
```

The interior cells are stored near the end of page 2. For example:

```text
000003a0: 12 81 5a 00 00 00 11 81 4d 00 00 00 10 81 40 00
000003b0: 00 00 0f 81 32 00 00 00 0e 81 25 00 00 00 0d 81
000003c0: 18 00 00 00 0c 81 0b 00 00 00 0b 7e 00 00 00 0a
000003d0: 70 00 00 00 09 62 00 00 00 08 54 00 00 00 07 46
000003e0: 00 00 00 06 38 00 00 00 05 2a 00 00 00 04 1c 00
000003f0: 00 00 03 0e
```

Example decode at file offset `0x03ef`:

```text
00 00 00 03 0e
```

- `00 00 00 03` = left child page 3
- `0e` = separator rowid 14

## Worked Lookup Example: Find `id = 150`

Because `id INTEGER PRIMARY KEY` is the rowid, lookup uses rowid 150.

Step 1: Start at root page 2.

Root separator rowids:

```text
14, 28, 42, 56, 70, 84, 98, 112, 126, 139, 152, 165, 178, 192, 205, 218
```

Step 2: Compare 150 with separators.

```text
150 > 139
150 <= 152
```

Therefore the lookup follows root cell 10 to left child page 13.

From the parsed page:

```text
cell 10; ptr=0x01bb; file=0x03bb; left_child=page 13; separator_rowid=152
```

Step 3: Go to page 13.

```text
page 13 file offset = (13 - 1) * 512 = 6144 = 0x1800
```

Page 13 is a leaf table B-tree page containing rowids 140-152.

Relevant parsed cell:

```text
cell 10; ptr=0x0081; file=0x1881; rowid=150; payload=32
```

So the row with `id = 150` begins at file offset `0x1881`.

## Address Calculation Rules Used

To move from page-relative addresses to real file addresses:

```text
page_start = (page_number - 1) * page_size
file_offset = page_start + page_relative_offset
```

Example for page 13, cell pointer `0x0081`:

```text
page_start = (13 - 1) * 512 = 0x1800
cell_file_offset = 0x1800 + 0x0081 = 0x1881
```

Example for page 2, cell pointer `0x01bb`:

```text
page_start = (2 - 1) * 512 = 0x0200
cell_file_offset = 0x0200 + 0x01bb = 0x03bb
```

## Conclusion

This lab shows that a SQLite table is stored as a B-tree. The database header gives the page size and page count. The schema table on page 1 points to root page 2 for the `students` table. Page 2 is an interior table B-tree page with child pointers to leaf pages 3 through 19. Each leaf page stores table records ordered by rowid. A lookup starts at the root page, compares the target rowid with separator rowids, follows the correct child pointer, and finally scans the target leaf page's cell pointer array to find the exact record offset.

## Important Observations

1. SQLite stores table data using B-tree pages.
2. Metadata is also stored inside the database as the `sqlite_schema` table.
3. The SQLite file header stores the page size and total page count.
4. Page 2 is the root page of the `students` table.
5. Page 2 is an interior B-tree node because the table was large enough to split across many pages.
6. Interior B-tree cells store child page pointers and separator rowids.
7. Leaf B-tree pages store the actual student records.
8. Cell pointer arrays allow SQLite to locate variable-length records inside a page.
9. File addresses can be calculated using `(page_number - 1) * page_size + page_relative_offset`.
10. The database can be manually navigated using only `xxd` output and SQLite page-format rules.
